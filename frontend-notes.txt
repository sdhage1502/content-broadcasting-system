2. AUTHENTICATION FLOW
----------------------
- Backed by Firebase Authentication (email/password). No bcrypt, no
  localStorage tokens, no custom JWT — the Firebase SDK handles ID tokens
  and attaches them to every Firestore call automatically.

- Login → authService.login(email, password):
    1. signInWithEmailAndPassword(auth(), email, password)
    2. Read profile doc users/{uid} → return public User shape.

- Self-service signup → authService.signup({ name, email, password, role }):
    1. createUserWithEmailAndPassword on the PRIMARY app
       (no Firestore pre-flight read — visitors aren't yet authenticated
       and Firestore rules require auth, so the duplicate check would
       itself fail with permission-denied; Firebase Auth rejects
       duplicates natively with `auth/email-already-in-use`).
    2. setDoc(users/{uid}, {...})

- Principal-driven teacher provisioning → authService.createTeacher(...):
    1. createSecondaryAuth() spins up a NAMED secondary Firebase app
       with its own Auth + Firestore so createUserWithEmailAndPassword
       doesn't replace the principal's session on the primary app.
    2. setDoc(users/{uid}) is performed via the SECONDARY Firestore
       so request.auth.uid == userId on the rules side, which satisfies
       `allow create: if request.auth.uid == userId`.
    3. dispose() signs out and deleteApp()s the secondary app in
       a finally block.

- AuthContext subscribes via onAuthStateChanged and reactively loads
  the matching profile from Firestore. ProtectedRoute waits for this
  hydration before rendering or redirecting.


3. ROLE-BASED ACCESS
--------------------
- ProtectedRoute (components/common/ProtectedRoute.tsx) wraps every
  authenticated route and:
    * redirects to /login if no user,
    * redirects to ROLE_HOME[user.role] if role mismatches,
    * renders LoadingState while AuthContext hydrates.

- Client-side scoping:
    * Teachers only ever see their own content — useContentList injects
      `teacherId: user.teacherId` automatically when role === "teacher".
    * Principals see all content unless they explicitly filter.

- Server-side enforcement (firestore.rules):
    * users/{userId}: read=auth, create=self-uid, update=self-or-principal.
    * contents/{id}: read=approved-public-or-auth; create=teacher-self;
      update=own-pending OR principal; principals get full control.


4. SERVICE LAYER (REPLACEABLE API DESIGN)
-----------------------------------------
- src/services/* are the ONLY files that call into firebase/firestore
  or hit /api/upload. Components and hooks consume them.
- Each service returns plain DTOs from src/types/index.ts.
- Swapping Firebase for a real REST/GraphQL backend is a one-file change
  per service — the contract (function names, args, return shape) stays.
- The same applies to Cloudinary: ContentUploadForm calls
  contentService.create(), which posts to /api/upload, which talks to
  Cloudinary. The form has no idea Cloudinary exists.


5. CLOUDINARY USAGE
-------------------
- Server-only credentials (CLOUDINARY_API_KEY, CLOUDINARY_API_SECRET) are
  read inside src/lib/cloudinary.ts and used by /api/upload to do a
  signed upload_stream. The secret never reaches the browser.
- The browser-side variable NEXT_PUBLIC_CLOUDINARY_CLOUD_NAME is needed
  by next-cloudinary's <CldImage> in ContentPreview and LiveContent for
  on-the-fly format/quality optimisation. Both env vars must be set.
- Files are uploaded to `contents/<teacherId>/<timestamp>_<safeName>`,
  with resource_type=image for images and raw for PDFs/DOCX.
- After upload we persist { fileUrl, filePublicId, fileVersion, fileType }
  on the Firestore content doc so <CldImage src={publicId}> can rebuild
  optimised URLs without trusting client input.
- next.config.ts whitelists res.cloudinary.com under images.remotePatterns
  so plain <Image> usage from Next is also fine.


6. ERROR HANDLING & VALIDATION
------------------------------
- All forms use react-hook-form + Zod (login, signup, upload, reject).
  Inline `<p class="text-destructive">` messages render the first Zod
  issue per field.
- The Teachers > Add teacher dialog uses a small bespoke validator
  (name length, email regex, password ≥ 6) so errors render inline AND
  a toast surfaces server-side failures.
- lib/utils.ts → getErrorMessage(e, fallback) maps Firebase codes to
  friendly strings:
      auth/invalid-credential       → "Invalid email or password."
      auth/email-already-in-use     → "An account with this email already exists."
      auth/weak-password            → "Password is too weak — use at least 6 characters."
      auth/too-many-requests        → "Too many attempts. Please wait a few minutes…"
      auth/network-request-failed   → "Network error. Check your connection…"
      permission-denied             → "You don't have permission to perform this action."
      …
  Every async surface (login, signup, upload, approve, reject, create
  teacher) funnels errors through this helper before toasting.


7. FIRESTORE SCHEMA
-------------------
users/{uid}    (uid = Firebase Auth uid)
  email:        string             // lowercased
  name:         string
  role:         "teacher" | "principal"
  teacherId?:   string | null      // present only for teachers
  createdAt:    Timestamp

contents/{autoId}
  title, subject, description?:     strings
  fileUrl:                          string       // Cloudinary secure_url
  fileType:                         string       // MIME
  filePublicId, fileVersion:        Cloudinary handle for <CldImage>
  startTime, endTime:               Timestamp
  rotationDuration:                 number (minutes)
  status:                           "pending" | "approved" | "rejected"
  rejectionReason?:                 string
  teacherId, teacherName:           strings
  createdAt, reviewedAt?:           Timestamp


8. LIVE PAGE (REALTIME)
-----------------------
- liveService.subscribe(teacherId, cb, onError) attaches a Firestore
  onSnapshot to `where teacherId==X AND status=="approved"`.
- The current time window (startTime ≤ now ≤ endTime) is filtered
  client-side after each snapshot — Firestore can't range-query two
  fields at once.
- Approvals/rejections propagate to the live screen in near-realtime
  with NO polling.
- LiveContent rotates through items via a setTimeout keyed off each
  item's rotationDuration. Empty state ("No content available") is
  rendered when nothing matches.
- /live/:teacherId requires NO authentication.


9. STATE MANAGEMENT
-------------------
- Auth: React Context (AuthContext), role-aware, hydrated by
  onAuthStateChanged.
- Content data: useContentList + useStats hooks over the services
  layer. Each returns { data, loading, error, refetch }. Params are
  memoised with useMemo so React doesn't re-fetch on every render.
- Live data: a single Firestore subscription inside LiveContent with
  a cleanup callback.
- Form state: react-hook-form + Zod.
- Theme: next-themes + header ThemeToggle (light/dark/system).


10. INDEXES & SORTING
---------------------
content.service.ts intentionally omits server-side orderBy and sorts
client-side, so principal/teacher list pages never hit the
"requires an index" error during development. firestore.indexes.json
ships the composite indexes you'd want when you flip back to
server-side sort:

    contents (teacherId asc, status asc, createdAt desc)
    contents (teacherId asc, createdAt desc)
    contents (status asc, createdAt desc)


11. ASSUMPTIONS
---------------
- Firebase project has Firestore + Authentication (email/password) on.
- Files ≤ 10 MB; allowed: JPG, PNG, GIF, WebP, PDF, DOCX.
  Server-side validation re-checks these in /api/upload before signing.
- The live page only displays IMAGE content (PDFs/DOCX are still
  uploadable and downloadable from teacher/principal views, just not
  part of the broadcast loop).
- Tailwind v4 tokens defined in globals.css via @theme inline.
- Next.js 16 specifics:
    * Dynamic route pages receive params as a Promise —
      /live/[teacherId]/page.tsx awaits it.
    * middleware.ts is intentionally not used; auth is gated
      client-side via ProtectedRoute (Firebase auth state lives on
      the client).


12. HOW TO RUN
--------------
Prerequisites: Node 18.18+ (20+ recommended), a Firebase project
(Firestore + Authentication), and a Cloudinary account.

  npm install
  cp .env.example .env.local        # fill Firebase + Cloudinary keys
  firebase deploy --only firestore:rules,firestore:indexes \
                  --project <your_project_id>
  npm run dev                       # http://localhost:3000

Quick walkthrough:
1. Visit / → redirected to /login.
2. Sign in as principal max@yc.com / max123, OR teacher@yc.com /
   teacher123, OR /signup as a new teacher.
3. Principal > Teachers → Add teacher (uses secondary Firebase app
   so the principal stays signed in). Open "View content" on each
   card to see that teacher's submissions.
4. Teacher > Upload → drag-drop an image → submit. Status starts
   as pending.
5. Principal > Pending Approval → preview, approve, or reject
   (reason ≥ 5 chars).
6. Open /live/<teacherId> in another window to see approved +
   in-window items rotate in realtime.
